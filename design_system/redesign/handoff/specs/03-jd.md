# Step 3 — JD Review

Maps to `frontend/src/components/workspace/JDReview.tsx`. Same props/callbacks as the original handoff; mirrors the Step 1 visual pattern (intake card → parsed hero → sections).

The shipped surface follows the original sketch closely. The main additions during the build:

- The JD body sections render whatever headings the parser returns, but the parser quality lifted to ~0.99 on a 15-fixture set after the LLM-first hybrid landed (vs ~0.78 deterministic baseline) so the sections users see are noticeably tidier than the prototype's mock data.
- When a job is selected in Step 2, the JD prefills with the cached posting metadata (description text + summary + location + URL) — no extra fetch round trip, the cached-jobs index already has the description bytes.

## Mode toggle

A three-way toggle at the top of the import card: `Paste text | From URL | File`. State is local to `JDReview`; switching modes does not discard the other modes' draft state.

## Selected-from-Step-2 prefill

If the user selected a job in Step 2 and clicked **Review role**, Step 3 mounts with the import card collapsed and the parsed JD hero already populated:

- Source-of-truth is the `cached_jobs` row, not a fresh upstream fetch — the cache already has the full description bytes from the latest refresh cycle, so prefill is one Supabase read away.
- The "Re-parse" button on the hero re-runs the parser against the same description text without re-importing.
- If the user wants to swap to a different JD, clicking the import card's mode toggle re-opens the intake surface; the active job (and its parsed JD) is cleared from workspace state.
- For sources we don't cache (e.g., a pasted URL outside the four supported ATSes), the prefill path falls back to `resolveJobUrl` with the same downstream behaviour.

## Behavior preservation

- Paste mode: parse on `[Parse JD]` click (not on blur — keeps the gesture explicit).
- URL mode: `resolveJobUrl` then auto-parse.
- File mode (drag-drop into the import card or file picker): `uploadJobDescriptionFile`.
- Hard/soft skill chips render parser output verbatim — engineer must NOT reorder, dedupe, or filter them.
- The original handoff's "Match score 91" hero stat doesn't ship — the workspace shows match signals (hard skills count, soft skills count, years) but doesn't compute a single composite score in the JD hero anymore. The fit_service produces a numeric score internally; it's only surfaced in the analysis run output.

## Section rendering rule

The JD body sections are dynamic. The parser returns a list of `{heading, items}` blocks. Render each as a `.b-jd-section` with the heading uppercased into the eyebrow slot and items as a bulleted list. Do not invent section titles — use exactly what the parser returns.

The deterministic parser is still the fallback path; the LLM hybrid (Tier-1 quality runner: 0.99 average across the 15-fixture set) is the default. Both produce the same `ParsedJobDescription` shape so the renderer doesn't care which one ran.

## States

Same pattern as Step 1: empty → loading → parsed → error.

| State | Visual |
|---|---|
| Empty (no JD yet) | Import card visible; hero hidden |
| Loading | Import card shows spinner; hero shows skeleton |
| Parsed | Import card collapses to "Loaded JD · [title]"; hero + summary + skills + body visible |
| Parse error | Notice panel above import card; import card stays open |
| Selected-from-Step-2 | Import card pre-collapsed; hero shows the cached posting; "Re-parse" available |
